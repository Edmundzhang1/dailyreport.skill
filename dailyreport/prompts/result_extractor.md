# 完成结果提取 Prompt

本文件用于从目标时间窗口内的 `Codex` 对话日志中提取“已完成且有意义”的结果。

## 目标

不要总结所有对话，只识别那些已经闭环、具有独立汇报价值、并且能够被日志证据支撑的完成结果。

## 提取对象

优先关注以下链路：

- 用户提出明确任务
- assistant 给出分析、实现、判断或结论
- 对话最终形成明确完成结果

## 提取步骤

1. 先按用户任务或工作目标划分候选片段。
2. 再判断这些片段是否形成了独立完成结果。
3. 对每个候选结果保留最小证据：
   - 任务目标的证据
   - 完成闭环的证据
4. 不满足准入条件的候选结果直接丢弃，不进入后续合并。

## 完成判定

只有当某个任务目标已经闭环，且不依赖额外动作才能成立时，才可判定为“已完成”。

以下情形通常可视为完成：

- 完成功能实现
- 完成问题定位
- 完成规则定义
- 完成方案选型
- 完成文档或规划书定稿
- 完成验证并形成明确结论

以下情形通常不视为完成：

- 仍在讨论
- 仍在实现中
- 只有候选方案，没有收敛
- 只做到一半
- 只有探索，没有明确结论
- 对话被中断且没有闭环

## 有意义判定

“有意义”不做固定类型枚举，按统一原则判断。

一个完成结果至少应满足以下判断方向之一：

- 对工作推进有明确贡献
- 对结果产出有独立价值
- 对决策收敛形成了明确结论
- 可以被独立写成一条工作总结

以下内容应排除：

- 纯过程噪声
- 重复确认
- 礼貌性往返
- 无结论讨论
- 只有措辞微调，没有新增结果

## 证据要求

每个候选完成结果都必须能回溯到日志中的明确证据。

至少保留两类证据中的一类或两类组合：

- 用户提出目标或确认目标
- assistant 明确给出完成、结论、定稿、定位、验证通过等闭环信号

不要凭感觉补全用户没有说过、日志里没有形成闭环的结果。

## 输出要求

本阶段的输出是内部候选结果集合，不是最终用户文本。

每个候选结果至少保留：

- 结果标题
- 结果摘要
- 完成证据
- 价值说明

不要在本阶段生成最终汇报措辞。
